## 1. 時序圖 (Sequence Diagrams)

### 1.1 登入流程

```plantuml
@startuml 登入時序圖
title 登入流程 — Supabase Auth + JWT + RLS

actor "使用者 (店長/員工)" as User
participant "React App\n(AuthContext)" as App
participant "Supabase Auth\n(GoTrue)" as Auth
participant "Supabase\nDatabase" as DB
participant "profiles\nTable" as Profiles

== 登入 ==
User -> App: 輸入 Email + Password\n點擊「登入」
App -> Auth: signInWithPassword(email, password)

alt 認證失敗
  Auth --> App: error "Invalid login credentials"
  App --> User: 顯示 "帳號或密碼錯誤"
else 認證成功
  Auth --> App: session { access_token, user }
  App -> App: onAuthStateChange 觸發

  == 讀取角色 ==
  App -> DB: SELECT * FROM profiles\nWHERE id = auth.uid()
  note right: anon key + JWT\nRLS 驗證中...

  DB -> DB: public.user_role()\n讀取 JWT app_metadata.role

  alt JWT 有 role claim
    DB --> App: { id, name, role, phone, business_id }
    App -> App: setUser({...})
    App --> User: 進入主畫面 (Sidebar + 今日預約)
  else JWT 無 role claim
    DB --> App: error 403 / empty
    App -> App: setLoading(false), user=null
    App --> User: 卡在 Loading 或回登入頁
  end
end

== 登出 ==
User -> App: 點擊「登出」
App -> Auth: signOut()
Auth --> App: session = null
App -> App: setUser(null)
App --> User: 返回登入頁面
@enduml
```

### 1.2 建立預約 (三維防撞)

```plantuml
@startuml 建立預約時序圖
title 建立預約 — 三步驟表單 + 三維防撞檢查

actor "員工 / 店長" as User
participant "NewAppointment\nPage" as Page
participant "Supabase\nREST API" as API
database "PostgreSQL" as DB

== Step 1: 選擇客戶與療程 ==
User -> Page: 輸入客戶姓名搜尋
Page -> API: GET /rest/v1/clients\n?name=ilike.%陳%
API --> Page: [{ id, name, phone, ... }]
Page --> User: 顯示搜尋結果 (電話後四碼)

User -> Page: 點選客戶 + 選擇療程
Page -> Page: formData.client_id = ...\nformData.treatment_id = ...
User -> Page: 點擊「下一步」

== Step 2: 安排時間、美容師、房間、儀器 ==
User -> Page: 選擇日期
User -> Page: 點選時間 (例如 10:00)
Page -> Page: 計算 end_time\n= start_time + treatment.duration_minutes

Page -> API: GET /rest/v1/appointments\n?appointment_date=eq....\n&status=neq.cancelled\n&start_time=lt.{endTime}\n&end_time=gt.{startTime}
API --> Page: [{ staff_id, room_id, equipment_id }]

Page -> Page: checkConflicts()\n比對選中的 staff/room/equipment\n是否出現在衝突列表中

alt 有衝突
  Page --> User: ⚠️ 該美容師/房間/儀器此時段已被佔用\n按鈕 disabled
else 無衝突
  Page --> User: 綠燈，可繼續
end

User -> Page: 選擇美容師 Amy + 房間 A + 儀器 #1
Page -> Page: 再次 checkConflicts()
User -> Page: 點擊「下一步」

== Step 3: 確認並儲存 ==
Page --> User: 顯示預約摘要\n(客戶/療程/時間/美容師/房間)
User -> Page: 點擊「儲存並發送通知」

Page -> API: POST /rest/v1/appointments\n{\n  client_id, staff_id, treatment_id,\n  room_id, equipment_id,\n  appointment_date, start_time, end_time,\n  status: 'confirmed',\n  created_by: auth.uid()\n}

alt 儲存成功
  API --> Page: 201 Created
  Page --> User: 跳轉回首頁，今日預約列表更新
else 資料庫錯誤 (例如同時間已被搶佔)
  API --> Page: error
  Page --> User: alert("儲存失敗: ...")
end
@enduml
```

### 1.3 預約 → 已出席 → 扣數 (核心防呆)

```plantuml
@startuml 扣數時序圖
title 核心流程：預約改為已出席 → 自動彈出扣數彈窗 → Transaction 原子扣減

actor "員工" as Staff
participant "DailyAppointments\nPage" as Page
participant "DeductionModal" as Modal
participant "Supabase\nRPC" as RPC
database "PostgreSQL\n(Transaction)" as DB

== 點擊預約卡片 ==
Staff -> Page: 在今日預約列表\n點擊某個待出席的預約卡片
Page -> Page: handleStatusChange(app)\nsetShowDeduction(true)

Page -> Modal: show = true\n傳入 appointment 物件

== 載入可用療程 ==
Modal -> API: GET /rest/v1/client_services\n?client_id=eq.{id}\n&remaining_sessions=gt.0\n&select=*, treatments(name)

API --> Modal: [{ id, treatment_id, remaining_sessions,\ntotal_sessions, treatments: { name } }]

Modal -> Modal: 預設勾選與預約同 treatment_id 的項目
Modal --> Staff: 顯示扣減確認單\n客戶名 + 療程列表 (Checkbox 多選)

== 員工手動勾選 ==
Staff -> Modal: 勾選要扣減的療程\n(可多選)
Modal -> Modal: toggleSelect(id)\nselectedIds = [...]

== 確認扣減 ==
Staff -> Modal: 點擊「✅ 確認扣減」

Modal -> RPC: deduct_service_from_appointment(\n  p_appointment_id,\n  p_service_ids,\n  p_payment_method\n)

RPC -> DB: BEGIN (Transaction 開始)

RPC -> DB: SELECT * FROM appointments\nWHERE id = p_appointment_id\nFOR UPDATE (鎖定行)

alt 預約已是 attended
  DB --> RPC: RAISE EXCEPTION "不可重複扣減"
  RPC --> Modal: error
  Modal --> Staff: 顯示錯誤訊息
else 預約已 cancelled
  DB --> RPC: RAISE EXCEPTION "已取消無法扣減"
else 正常
  loop 每個選中的 client_service
    RPC -> DB: 檢查 expiry_date 是否過期
    RPC -> DB: 檢查 remaining_sessions > 0

    RPC -> DB: UPDATE client_services\nSET remaining_sessions = remaining_sessions - 1

    RPC -> DB: INSERT INTO activity_log\n(扣減前後對照)

    RPC -> DB: 累計金額 v_total_amount
  end

  RPC -> DB: UPDATE appointments\nSET status = 'attended'

  RPC -> DB: INSERT INTO payment_transactions\n(關聯 appointment, 金額, 支付方式)

  RPC -> DB: INSERT INTO activity_log\n(扣數完成摘要)

  RPC -> DB: COMMIT
  RPC --> Modal: { success: true, transaction_id, total_amount }
  Modal -> Page: onClose() → fetchAppointments()
  Page --> Staff: 預約列表更新，該預約灰底 + 已出席標籤
end
@enduml
```

### 1.4 退款流程

```plantuml
@startuml 退款時序圖
title 退款流程 — 僅店長可執行

actor "店長" as Owner
participant "ClientDetail\nPage" as Page
participant "RefundModal\n(尚未實作)" as Modal
participant "Supabase\nRPC" as RPC
database "PostgreSQL\n(Transaction)" as DB

== 進入客戶詳情 ==
Owner -> Page: 打開客戶詳情頁\n右側看到已購療程庫存

Owner -> Page: 點擊某個療程的「退款」按鈕\n(目前尚未實作)

== 填寫退款原因 ==
Page -> Modal: show = true\n傳入 client_service
Modal --> Owner: 顯示退款表單\n• 退款金額\n• 回補次數\n• 🔴 退款原因 (必填)

Owner -> Modal: 填寫退款原因 + 金額\n點擊「確認退款」

== RPC 執行 ==
Modal -> RPC: refund_deduction(\n  p_client_service_id,\n  p_sessions_to_restore,\n  p_refund_amount,\n  p_reason\n)

RPC -> DB: 檢查 public.is_admin() = true

alt 非店長
  RPC --> Modal: RAISE EXCEPTION "權限不足"
else 店長
  RPC -> DB: BEGIN

  RPC -> DB: SELECT * FROM client_services\nWHERE id = ... FOR UPDATE

  RPC -> DB: UPDATE client_services\nSET remaining_sessions += restored,\n    status = 'refunded'

  RPC -> DB: INSERT INTO refunds\n(client_service_id, refunded_by,\n refund_amount, restored_sessions, reason)

  RPC -> DB: INSERT INTO activity_log\n(退款前後對照)

  RPC -> DB: COMMIT
  RPC --> Modal: { success: true, message: '退款完成' }
  Modal -> Page: onClose() → 重整客戶詳情
  Page --> Owner: 剩餘次數已回補，狀態顯示「已退款」
end
@enduml
```

### 1.5 每日結算鎖定

```plantuml
@startuml 每日結算時序圖
title 每日結算 — 自動加總 + 二次確認鎖定

actor "店長" as Owner
participant "DailySettlement\nPage" as Page
participant "Supabase\nREST API" as API
participant "Supabase\nRPC" as RPC
database "PostgreSQL" as DB

== 進入結算頁 ==
Owner -> Page: 打開「每日結算」
Page -> API: GET /rest/v1/payment_transactions\n?transaction_date=eq.{today}\n&select=*, clients(name), treatments(name)
API --> Page: [{ 交易紀錄 }]

Page -> API: GET /rest/v1/daily_settlements\n?settlement_date=eq.{today}
API --> Page: settlement || null

Page -> Page: 計算 totals:\ntotal, cash, card, transfer
Page --> Owner: 顯示四個金額卡片\n+ 交易明細 Table

alt 當日尚未鎖定
  Page --> Owner: 顯示「備註/備用金差異」輸入欄\n+「完成結算並鎖定 🔒」按鈕

  Owner -> Page: (可選) 輸入備註差異
  Owner -> Page: 點擊「完成結算並鎖定 🔒」

  == 二次確認 ==
  Page -> Page: setShowConfirm(true)
  Page --> Owner: Modal: "⚠️ 一旦鎖定將無法自行更改\n如需調整請聯繫總管理員"
  Owner -> Page: 點擊「確認鎖定」

  == RPC 鎖定 ==
  Page -> RPC: close_daily_settlement(\n  p_settlement_date,\n  p_difference_note\n)

  RPC -> DB: 檢查 public.is_admin()
  RPC -> DB: 檢查是否已 locked (防重複)

  RPC -> DB: SELECT SUM(amount) ...\nFROM payment_transactions\nWHERE transaction_date = ...

  RPC -> DB: UPSERT daily_settlements\nSET status = 'locked', locked_by = ..., locked_at = now()

  RPC -> DB: UPDATE payment_transactions\nSET settlement_id = ...\nWHERE transaction_date = ...

  RPC -> DB: INSERT INTO activity_log\n(結算鎖定紀錄)

  RPC --> Page: { success: true, settlement_id, total_amount }
  Page -> Page: 重整頁面
  Page --> Owner: 🟢 顯示「結算已鎖定」Badge\n所有輸入欄 disabled

else 當日已鎖定
  Page --> Owner: 🟢 「結算已鎖定」Badge\n唯讀模式
end
@enduml

### 1.6 店長退回已出席預約 (扣數撤銷)

```plantuml
@startuml 退回已出席時序圖
title 扣數撤銷 — 店長將 attended 退回 confirmed + 自動回補次數 + Log 全部不可刪

actor "店長" as Owner
participant "DailyAppointments\nPage" as Page
participant "Supabase\nRPC" as RPC
database "PostgreSQL\n(Transaction)" as DB

== 店長發現誤扣，決定退回 ==
Owner -> Page: 在今日預約列表中\n看到一個狀態為「已出席」的預約
Owner -> Page: 點擊該預約卡片\n(目前只有店長看到「退回」按鈕)

Page -> Page: 判斷 user.role === 'shop_owner'\n且 appointment.status === 'attended'
Page --> Owner: 顯示「🔙 退回已出席」按鈕\n(員工看不到此按鈕)

Owner -> Page: 點擊「退回已出席」

Page -> Page: setShowRevertConfirm(true)
Page --> Owner: 🔴 Modal: "⚠️ 退回已出席確認"\n"此操作將：\n1. 回補已扣減的療程次數\n2. 刪除對應的交易紀錄\n3. 預約狀態回到「已確認」\n\n所有操作將記錄在 activity_log 中\n(不可刪除)\n\n請輸入退回原因："

Owner -> Page: 填寫退回原因\n(必填，例如："客戶臨時取消，已溝通")

Owner -> Page: 點擊「確認退回」

== RPC 原子退回 ==
Page -> RPC: revert_attended_to_confirmed(\n  p_appointment_id,\n  p_reason\n)

RPC -> DB: BEGIN (Transaction)

' 權限檢查
RPC -> DB: 檢查 public.is_admin() = true

' 鎖定預約行
RPC -> DB: SELECT * FROM appointments\nWHERE id = p_appointment_id\nFOR UPDATE

alt 預約狀態 != attended
  RPC --> Page: ❌ RAISE EXCEPTION\n"只有已出席的預約可以退回"
else 狀態正確
  ' 查找該預約相關的所有 payment_transactions
  RPC -> DB: SELECT * FROM payment_transactions\nWHERE appointment_id = p_appointment_id

  loop 每筆交易
    ' 反查對應的 activity_log (deduct_service)
    RPC -> DB: SELECT target_id, details\nFROM activity_log\nWHERE action_type = 'deduct_service'\nAND details->>'appointment_id' = p_appointment_id

    ' 回補每個 client_service
    RPC -> DB: UPDATE client_services\nSET remaining_sessions = remaining_sessions + 1,\n    status = 'active'\nWHERE id = log_details->>'client_service_id'

    ' 標記該 payment_transaction 為 void
    RPC -> DB: UPDATE payment_transactions\nSET remarks = '[VOID] 店長退回: ' || p_reason\nWHERE id = tx.id

    ' 寫入退回 Log (不可刪)
    RPC -> DB: INSERT INTO activity_log\n(user_id, action_type, target_type, target_id, details)\nVALUES (\n  v_user_id,\n  'revert_deduction',\n  'client_service',\n  tx_svc_id,\n  jsonb_build_object(\n    'original_deduction_log_id', log.id,\n    'restored_sessions', 1,\n    'reason', p_reason,\n    'appointment_id', p_appointment_id\n  )\n)
  end

  ' 更新預約狀態回 confirmed
  RPC -> DB: UPDATE appointments\nSET status = 'confirmed',\n    remarks = COALESCE(remarks || ' | ', '') ||\n              '店長退回: ' || p_reason,\n    updated_at = now()\nWHERE id = p_appointment_id

  ' 寫入退回摘要 Log
  RPC -> DB: INSERT INTO activity_log\n(user_id, action_type, target_type, target_id, details)\nVALUES (\n  v_user_id,\n  'revert_appointment',\n  'appointment',\n  p_appointment_id,\n  jsonb_build_object(\n    'previous_status', 'attended',\n    'new_status', 'confirmed',\n    'reason', p_reason,\n    'voided_transactions', tx_ids\n  )\n)

  RPC -> DB: COMMIT
  RPC --> Page: { success: true, message: '已退回' }

  Page -> Page: fetchAppointments() 重整
  Page --> Owner: 該預約恢復為「已確認」\n可以重新標記出席 (重新扣數)
end
@enduml
```

### 1.7 退回後的重新扣數 (邊緣案例)

```plantuml
@startuml 退回後重新扣數時序圖
title 退回後重新扣數 — 確保不重複、不遺漏

actor "員工" as Staff
participant "Page" as Page
participant "RPC" as RPC
database "DB" as DB

== 退回後，預約回到 confirmed ==
Staff -> Page: 看到該預約狀態為「已確認」\n(客戶決定不取消了，照常來店)

Staff -> Page: 點擊預約卡片
Page -> Page: 檢查 appointment.status === 'confirmed'\n→ 正常彈出 DeductionModal

== 載入可用療程 ==
Page -> DB: SELECT * FROM client_services\nWHERE client_id = ...\nAND remaining_sessions > 0\nAND status = 'active'

note right: 因為之前退回時\n已回補 remaining_sessions\n且 status 改回 'active'\n所以療程重新出現在列表中

DB --> Page: [{ 已回補的療程 }]

Page -> Page: 預設勾選與預約同 treatment_id 的項目

Staff -> Page: 勾選療程 → 點擊「確認扣減」

== 重新扣數 ==
Page -> RPC: deduct_service_from_appointment(\n  p_appointment_id,\n  p_service_ids\n)

RPC -> DB: 檢查 appointment.status = 'confirmed'\n(上次退回已改回 confirmed，所以通過)

RPC -> DB: 正常扣減流程\n(扣次數 + 寫 payment + 寫 log)

RPC --> Page: { success: true }

note over DB
  activity_log 中現在有：
  1. 第一次 deduct_service (已回補)
  2. revert_deduction (店長退回)
  3. revert_appointment (狀態變更)
  4. 第二次 deduct_service (新扣數)
  
  🛡️ 全部 INSERT-ONLY，無法刪改
  形成完整審計軌跡
end note

Page --> Staff: 預約再次顯示「已出席」
@enduml
```

## 2. 狀態機圖 (State Machine Diagrams) — 修訂版

### 2.1 預約狀態機 (Appointment) — 含退回路徑

```plantuml
@startuml 預約狀態機v2
title 預約 (Appointment) 狀態機 v2 — 含店長退回路徑

[*] --> confirmed : 員工建立預約\n(三維防撞通過)

confirmed --> attended : 員工點擊預約卡片\n→ 扣數 Modal → 確認扣減\n(RPC: deduct_service_from_appointment)

confirmed --> cancelled : 員工/店長取消預約\n(記錄取消原因)

confirmed --> no_show : 員工標記為失約\n(Trigger 自動計算\nno_show_count)

no_show --> confirmed : 店長重新啟用\n(清除失約標記)

attended --> confirmed : 🔙 店長退回已出席\n(RPC: revert_attended_to_confirmed)\n• 回補所有已扣次數\n• payment_transactions 標記 VOID\n• appointment.remarks 記錄原因\n• activity_log 寫入退回紀錄\n🛡️ 所有 Log INSERT-ONLY

note right of attended
  🔙 退回路徑 (僅店長)
  attended → confirmed:
  1. 回補 client_services 次數
  2. payment_transactions → VOID
  3. activity_log 寫入退回原因
  4. 預約回到 confirmed，可重新扣數
  ⚠️ 原扣數 Log 仍然保留，不可刪
end note

note right of no_show
  ⚠️ 失約累計
  • Trigger 自動計算 history count
  • ≥2 次 → 前端應顯示警告標籤
  • 不自動阻擋預約
end note

attended --> [*]
cancelled --> [*]
@enduml
```

### 2.2 client_services 狀態機 — 含退回路徑

```plantuml
@startuml client_services狀態機v2
title 客戶已購療程庫存 (client_services) 狀態機 v2 — 含退回路徑

[*] --> active : 店長建立購買紀錄\n或 RPC: manual_grant_sessions()

active --> active : 每次扣減\nremaining_sessions - 1\n(但仍 > 0)

active --> expired : 扣減後 remaining_sessions = 0\n(自動設為 expired)

active --> expired : 手動過期處理\n(店長操作，尚未實作 UI)

active --> refunded : 店長執行退款\n(RPC: refund_deduction)\n• 回補次數\n• 寫入 refunds 表

refunded --> active : 🔙 如果退款是誤操作\n店長可重新啟用\n(尚未實作 UI)

active --> active : 🔙 店長退回已出席\n(RPC: revert_attended_to_confirmed)\n• remaining_sessions + 1\n• status 保持 active\n• 該預約回到 confirmed

expired --> [*]
refunded --> [*]

note right of active
  🔒 防弊設計
  • 前端不可直接 UPDATE remaining_sessions
  • 只能透過 SECURITY DEFINER RPC
  • 扣減 + 退回 全部寫入 activity_log
  • 即使退回，原扣數 Log 保留
end note
@enduml
```

## 5. 需要新增的 SQL：revert_attended_to_confirmed RPC

```sql
-- ============================================================
-- 8.6 revert_attended_to_confirmed() — 🔙 店長退回已出席
--     將 attended 退回 confirmed，自動回補所有已扣次數
--     • payment_transactions 標記 VOID (不刪除)
--     • client_services 回補次數
--     • activity_log 寫入完整退回紀錄
--     • 所有操作在單一 Transaction 中完成
-- ============================================================
CREATE OR REPLACE FUNCTION revert_attended_to_confirmed(
  p_appointment_id UUID,
  p_reason         TEXT
)
RETURNS JSONB AS $$
DECLARE
  v_user_id     UUID;
  v_appt            public.appointments%ROWTYPE;
  v_log         RECORD;
  v_restored    INT := 0;
  v_tx_ids      UUID[] := '{}';
BEGIN
  v_user_id := auth.uid();

  -- 🔴 權限檢查：僅店長
  IF NOT public.is_admin() THEN
    RAISE EXCEPTION '權限不足：只有店長可以退回已出席的預約';
  END IF;

  -- 鎖定預約行
  SELECT * INTO v_appt
  FROM public.appointments
  WHERE id = p_appointment_id
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION '找不到預約紀錄';
  END IF;

  -- 🔴 只有 attended 可以退回
  IF v_appt.status != 'attended' THEN
    RAISE EXCEPTION '只有已出席的預約可以退回，目前狀態：%', v_appt.status;
  END IF;

  -- 收集所有由此次預約產生的 payment_transactions
  FOR v_log IN
    SELECT al.target_id AS client_service_id,
           (al.details->>'after_remaining')::INT AS after_remaining,
           (al.details->>'before_remaining')::INT AS before_remaining
    FROM public.activity_log al
    WHERE al.action_type = 'deduct_service'
      AND al.details->>'appointment_id' = p_appointment_id::TEXT
  LOOP
    -- 回補 client_services 次數
    UPDATE public.client_services
    SET remaining_sessions = remaining_sessions + 1,
        status = 'active'
    WHERE id = v_log.client_service_id;

    IF FOUND THEN
      v_restored := v_restored + 1;

      -- 寫入退回明細 Log
      INSERT INTO public.activity_log (user_id, action_type, target_type, target_id, details)
      VALUES (
        v_user_id,
        'revert_deduction',
        'client_service',
        v_log.client_service_id,
        jsonb_build_object(
          'appointment_id', p_appointment_id,
          'restored_sessions', 1,
          'before_remaining', COALESCE(v_log.after_remaining, 0),
          'after_remaining', COALESCE(v_log.after_remaining, 0) + 1,
          'reason', p_reason
        )
      );
    END IF;
  END LOOP;

  -- 標記所有關聯的 payment_transactions 為 VOID
  WITH updated AS (
    UPDATE public.payment_transactions
    SET remarks = COALESCE(remarks || ' | ', '') || '[VOID] 店長退回: ' || p_reason
    WHERE appointment_id = p_appointment_id
    RETURNING id
  )
  SELECT array_agg(id) INTO v_tx_ids FROM updated;

  -- 更新預約狀態回 confirmed
  UPDATE public.appointments
  SET status = 'confirmed',
      remarks = COALESCE(remarks || ' | ', '') || '🔙 店長退回(' || now()::DATE::TEXT || '): ' || p_reason,
      updated_at = now()
  WHERE id = p_appointment_id;

  -- 寫入退回摘要 Log
  INSERT INTO public.activity_log (user_id, action_type, target_type, target_id, details)
  VALUES (
    v_user_id,
    'revert_appointment',
    'appointment',
    p_appointment_id,
    jsonb_build_object(
      'previous_status', 'attended',
      'new_status', 'confirmed',
      'reason', p_reason,
      'restored_services_count', v_restored,
      'voided_transaction_ids', v_tx_ids
    )
  );

  RETURN jsonb_build_object(
    'success', true,
    'restored_count', v_restored,
    'voided_transactions', v_tx_ids,
    'message', '已退回 ' || v_restored || ' 筆療程扣減，預約回到「已確認」'
  );

EXCEPTION WHEN OTHERS THEN
  RAISE;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER SET search_path = '';
```
