# **美容院管理雲端系統 (PWA) – 專案背景、需求與技術規範 v2.0**

## **1. 專案目標與背景**

* **使用者** ：一間本地中小型美容院，現時完全依賴紙本、WhatsApp 及人手記憶管理預約、療程次數及收款。
* **核心目標** ：建立一個**零營運成本、免安裝、方便店長記錄**的雲端管理系統，取代所有紙本作業。
* **使用者裝置** ：店內只有一部 iPad ，系統需在 Safari / Chrome 上流暢運行，並能加入主畫面作為 PWA。
* **成本限制** ：每月經常性開支必須為 HK$0（利用免費 tiers）。

## **2. 技術棧 (Tech Stack) – 已選定，不可更改**

| 層級          | 技術                                         | 用途                                   |
| ------------- | -------------------------------------------- | -------------------------------------- |
| 前端          | React (Vite) + Tailwind CSS                  | PWA，部署在 Vercel（免費 Hobby 計劃）  |
| 後端 / 資料庫 | Supabase                                     | PostgreSQL 資料庫、即時訂閱、Auth、RLS |
| 域名          | 免費域名 (.us.kg 或 Vercel 自帶 .vercel.app) | 對外存取                               |
| DNS / SSL     | Cloudflare (Free)                            | 管理 DNS 及強制 HTTPS                  |
| 檔案儲存      | Supabase Storage (Free 1GB)                  | 儲存療程前後照片（未來功能）           |

## **3. 權限架構**

系統必須內置三種角色，透過 Supabase Auth + Custom Claims 實現，前端及後端同時管制。

1. **IT 管理員 (開發者)**
   * 擁有 Supabase 後台直接存取權，不參與日常營運，僅作維護及數據修復。
2. **店長 (老闆)**
   * App 內 99% 最高權限：可增減客戶、療程、價錢、退款、手動贈送次數、查看所有報表。
   * **唯一限制** ：無法修改 / 刪除「系統活動日誌 (Log)」。
3. **一般員工**
   * 僅能查看 Booking、幫客戶做療程消耗次數、新增客戶備註。
   * **紅線** ：嚴禁修改療程價格、手動新增 / 贈送次數、查看總營業額報表，亦無法刪除任何資料。

## **4. 核心設計理念（v2 — 緩存模式）**

### 4.0.1 「收錢」唔係「付款」
店長端所有操作係「向客戶收錢」，唔係客戶「付款」。用詞反咉角色：店長係主動收錢嗰方。

### 4.0.2 「緩存」(Buffer) 概念
任何未完全完成嘅狀態 = 緩存：
- 做完服務但未收足錢 → 緩存
- 收咗錢但未做完服務 → 緩存
- 兩者對於收入角度嚟講同樣係「未結清」

只有**做晒所有次數 + 收足全額**先係「確定收入」。

### 4.0.3 「訂單」為本
每一筆 client_service = 一個訂單（Order）。訂單有：
- `total_price` — 總額
- `total_sessions` — 總次數
- `remaining_sessions` — 剩餘次數
- 已收金額 — 從 payment_transactions SUM 返嚟

收入記錄頁面 = 訂單記錄列表（唔係會計帳）。

### 4.0.4 「分期付款」
Package 總額可以分多次收取。每次收幾多錢，店長自由輸入。系統唔自動計算「應收」— 店長自己決定今次收幾多。

### 4.0.5 「治療項目 = 餐單，訂單 = 落單」
治療項目（treatments）係基礎餐單（有預設價格、時長）。但實際落單（create client_service）時店長可以自由調整金錢同次數（有人多啲、有人少啲）。所以 client_service 係「訂單」，唔係死板嘅套餐。

## **5. 功能需求 — 完整列表**

以下功能必須在第一階段全部實現，無一遺漏。

### **5.1 客戶管理**

* 欄位：姓名、電話、會員編號（系統自動生成）、備註、 **客戶來源標籤** （IG廣告 / 朋友介紹 / 街客等）、特殊敏感標記。
* 搜尋：支援電話或姓名搜尋，結果 **強制顯示電話後四碼及會員編號** ，防止同名同姓點錯。
* 警示：若客戶被標記為「特殊/敏感」，在開啟其預約單或收錢單時，系統強制彈出 Popup 提示。

### **5.2 療程管理**

* 店長可建立療程庫：名稱、分類、單次價格、套票總次數（作為預設餐單）。
* 客戶購買紀錄（訂單）：清楚列出該客戶已購買的所有療程、總次數、剩餘次數、總額、已收金額。
* 每張訂單可以自由調整次數同金額（唔跟死餐單預設值）。
* 三層防禦停用機制：有人用緊 → 顯示邊個用緊 + 封鎖停用；有歷史但冇活躍庫存 → 軟停用；冇人用過 → 可刪除。

### **5.3 預約管理（三維防撞）**

* 日曆 / 列表視圖顯示所有預約。
* 建立預約時，系統同時檢查「 **美容師 + 房間 + 儀器** 」三個維度的時段是否可用，有衝突即封鎖該時段。
* 支援員工排班設定（店長設定上班時間、休假），非辦公時段自動隱藏。

### **5.4 預約狀態與收錢連動**

* 預約狀態：已確認、已出席、取消、失約（失約 2 次自動標示警告）。
* 當員工將預約改為「已出席」時，系統 **自動彈出收錢確認單** ，列出該客戶所有可消耗療程，由員工手動勾選並確認，系統才寫入記錄。此舉防止忘記消耗次數，也防止誤扣。

### **5.5 收入紀錄（訂單記錄）**

* 兩個框架顯示：
  - 🟡 **緩存中** — 未完全做完服務 / 未收足錢嘅訂單
  - 🟢 **確定收入** — 做完 + 收足錢嘅訂單
* 下方列表每一行為一筆收錢動作（來自治療訂單），記錄：客戶、療程、收款方式、金額、完成度、日期。
* 分期付款顯示「已收 / 總額」進度。
* 唔需要「每日結算鎖定」— 收入記錄純粹係方便展示，唔鎖。

### **5.6 退款**

* 店長可對任何 client_service 執行退款。
* **自由輸入**：店長自己決定退幾多錢、回補幾多次數。
* 退款方式下拉：現金/信用卡/轉賬/其他。
* 退款原因必填。
* 退款後 client_service 狀態 = `refunded`，記錄寫入 refunds 表 + activity_log。

### **5.7 系統活動日誌 (Log)**

* 記錄所有關鍵操作：誰、何時、做了什麼。
* **技術要求** ：在 Supabase 資料庫層面，將 Log 表設定為 **INSERT-ONLY**（利用 RLS 禁止 UPDATE / DELETE）。

### **5.8 沉睡客篩選**

* 系統自動列出超過 90 天無預約紀錄的客戶，供店長跟進。

### **5.9 資料匯出**

* 一鍵匯出客戶名單、銷售紀錄為 Excel 檔案。

## **6. 用詞規範（v2 更新）**

| 舊詞（v1，已廢棄）| 新詞（v2）| 原因 |
|------|------|------|
| 扣減 / 扣數 | 消耗次數 / 收錢 | 店長端主動收錢，唔係被動扣減 |
| 每日結算 | 收入紀錄 | 純展示，唔鎖定唔對帳 |
| 今日收入 | 緩存概覽 + 確定收入 | 分開緩存同確定，唔混淆 |
| 支付 | 收錢 | 店長收錢，唔係客戶付款 |
| DeductionModal | PaymentModal / 收錢 Modal | 術語統一 |
| deduct_service | consume_service / collect_payment | RPC 層面（日後 migration） |

## **7. 資料庫設計重點**

* 核心表：`clients`, `treatments`, `client_services` (訂單), `appointments`, `rooms`, `equipment`, `profiles`, `staff_schedules`, `activity_log`, `payment_transactions`, `refunds`。
* `client_services` 係訂單概念：有 `total_price`, `total_sessions`, `remaining_sessions`, `unit_price`。
* `payment_transactions` 記錄每次收錢：有 `amount`, `amount_received`, `payment_method`。
* Log 表必須有 `id`, `user_id`, `action_type`, `target_type`, `target_id`, `details (JSONB)`, `created_at`，並設定 RLS: `USING (false)` for update/delete，僅允許 insert。
* 所有牽涉金錢或次數變動的操作，必須使用 Supabase Database Functions 或 Transactions，確保原子性。

## **8. 開發階段策略**

* UI / UX 必須為 iPad 觸控優化（按鈕夠大、流程簡單），PWA 離線緩存基本頁面。
