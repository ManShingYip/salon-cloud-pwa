# **美容院管理雲端系統 (PWA) – 專案背景、需求與技術規範 v1.0**

## **1. 專案目標與背景**

* **使用者** ：一間本地中小型美容院，現時完全依賴紙本、WhatsApp 及人手記憶管理預約、療程次數及收款。
* **核心目標** ：建立一個**零營運成本、免安裝、防員工走漏洞**的雲端管理系統，取代所有紙本作業。
* **使用者裝置** ：店內只有一部 iPad ，系統需在 Safari / Chrome 上流暢運行，並能加入主畫面作為 PWA。
* **成本限制** ：每月經常性開支必須為 HK$0（利用免費 tiers），僅需支付可選的 WhatsApp API 費用（未來或加）。

## **2. 技術棧 (Tech Stack) – 已選定，不可更改**

| 層級          | 技術                                         | 用途                                   |
| ------------- | -------------------------------------------- | -------------------------------------- |
| 前端          | React (Vite) + Tailwind CSS                  | PWA，部署在 Vercel（免費 Hobby 計劃）  |
| 後端 / 資料庫 | Supabase                                     | PostgreSQL 資料庫、即時訂閱、Auth、RLS |
| 域名          | 免費域名 (.us.kg 或 Vercel 自帶 .vercel.app) | 對外存取                               |
| DNS / SSL     | Cloudflare (Free)                            | 管理 DNS 及強制 HTTPS                  |
| 檔案儲存      | Supabase Storage (Free 1GB)                  | 儲存療程前後照片（未來功能）           |

## **3. 權限架構 – 三層防弊**

系統必須內置三種角色，透過 Supabase Auth + Custom Claims 實現，前端及後端同時管制。

1. **IT 管理員 (開發者)**
   * 擁有 Supabase 後台直接存取權，不參與日常營運，僅作維護及數據修復。
2. **店長 (老闆)**
   * App 內 99% 最高權限：可增減客戶、療程、價錢、退款、手動贈送次數、查看所有報表。
   * **唯一限制** ：無法修改 / 刪除「系統活動日誌 (Log)」。
3. **一般員工**
   * 僅能查看 Booking、扣減療程次數、新增客戶備註。
   * **紅線** ：嚴禁修改療程價格、手動新增 / 贈送次數、查看總營業額報表，亦無法刪除任何資料。

## **4. 功能需求 – 完整列表（第一階段 MVP）**

以下功能必須在第一階段全部實現，無一遺漏。

### **4.1 客戶管理**

* 欄位：姓名、電話、會員編號（系統自動生成）、備註、 **客戶來源標籤** （IG廣告 / 朋友介紹 / 街客等）、特殊敏感標記。
* 搜尋：支援電話或姓名搜尋，結果 **強制顯示電話後四碼及會員編號** ，防止同名同姓點錯。
* 警示：若客戶被標記為「特殊/敏感」，在開啟其預約單或扣數單時，系統強制彈出 Popup 提示。

### **4.2 療程管理**

* 店長可建立療程庫：名稱、分類、單次價格、套票總次數。
* 客戶購買紀錄：清楚列出該客戶已購買的所有療程、總次數、剩餘次數。
* 扣減時若客人有多個同類組合，系統須讓員工明確剔選要扣除哪一項， **嚴禁全自動盲目扣減** 。

### **4.3 預約管理（三維防撞）**

* 日曆 / 列表視圖顯示所有預約。
* 建立預約時，系統同時檢查「 **美容師 + 房間 + 儀器** 」三個維度的時段是否可用，有衝突即封鎖該時段。
* 支援員工排班設定（店長設定上班時間、休假），非辦公時段自動隱藏。

### **4.4 預約狀態與扣數連動 (防呆設計)**

* 預約狀態：已確認、已出席、取消、失約（失約 2 次自動標示警告）。
* 當員工將預約改為「已出席」時，系統 **自動彈出扣數確認單** ，列出該客戶所有可扣減療程，由員工手動勾選並確認扣減，系統才真正寫入扣數記錄。此舉防止忘記扣數，也防止誤扣。

### **4.5 每日一鍵結算板**

* 自動加總當日所有已完成療程的收入，按支付方式（現金、信用卡、轉賬等）分類顯示。
* **必須包含「備註 / 備用金差異」輸入欄位** ，讓員工填寫找錯錢或實收差異，確保帳面與實體現金一致。

### **4.6 退款與過期處理**

* 專屬介面讓店長處理退款或套票過期。
* 執行時需填寫原因，系統自動扣減當日收入（退款）、清零剩餘次數（過期），並寫入 **不可刪改的 Log** 。

### **4.7 系統活動日誌 (Log) – 絕對防弊**

* 記錄所有關鍵操作：誰、何時、做了什麼（例如「店長A 贈送 客戶B 激光療程 3 次」）。
* **技術要求** ：在 Supabase 資料庫層面，將 Log 表設定為  **INSERT-ONLY** （利用 RLS 禁止 UPDATE / DELETE），任何人（包括持有資料庫密碼者）都無法從後台篡改紀錄。

### **4.8 沉睡客篩選**

* 系統自動列出超過 90 天無預約紀錄的客戶，供店長跟進。

### **4.9 資料匯出**

* 一鍵匯出客戶名單、銷售紀錄為 Excel 檔案。

## **5. 資料庫設計重點（給 AI 的提示）**

* 核心表：`customers`, `treatments`, `customer_treatments` (購買紀錄與剩餘次數), `appointments`, `rooms`, `devices`, `staff`, `staff_schedules`, `logs`, `daily_settlements`, `refunds`。
* Log 表必須有 `id`, `user_id`, `action`, `details (JSONB)`, `created_at`，並設定 RLS: `USING (false)` for update/delete，僅允許 insert。
* 所有牽涉金錢或次數變動的操作，必須使用 Supabase Database Functions 或 Transactions，確保原子性。

## **6. 開發階段策略**

* **現在只做第一階段 MVP** ，必須覆蓋以上全部功能。
* UI / UX 必須為 iPad 觸控優化（按鈕夠大、流程簡單），PWA 離線緩存基本頁面。
* 完成後測試 1-2 星期，再討論是否加入第二階段（客戶自助查詢、WhatsApp 提醒等）。
